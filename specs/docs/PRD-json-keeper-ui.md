## 0. 给原型工具的硬约束（先读这节，违反即作废）

| 约束 | 说明 | 为什么 |
| --- | --- | --- |
| **交付形态 = Chrome 扩展** | 不是网站、不是桌面 App。三个界面各有物理尺寸限制，见 §2 | 画成网页则全错 |
| **零网络** | 不许出现登录、账号、云同步、分享链接、协作、"上传到…"、社交按钮 | 产品承诺写在状态栏与商店页；有网络即失去立身之本 |

| **全部计算在本地** | 新功能（查询/Schema/Diff）均为本地计算，不得设计成"请求中…"的远程调用 | 见 §5 |
| **深浅双主题** | 每个界面都要出浅色和深色两版 | 现有产品已支持手动 auto/☀/☾ |

## 1. 产品是什么

**一句话**：Chrome 扩展，把 JSON 变得可读、可复制、**可信**。

**它对标谁**：JSONVue（老牌 JSON 查看器）。JSON Keeper 是针对其差评根因的重写：

| JSONVue 的痛点（差评数） | 根因 | JSON Keeper 的答案 |
| --- | --- | --- |
| "用不了/不知道怎么用"（44） | 纯页面转换器，无 popup、无粘贴框、无入口 | **popup 粘贴框** + 独立 viewer 页 |
| 吐不回合法 JSON（25） | 复制藏在右键菜单，切不回原文 | 工具栏上**可见的 Copy / Raw 按钮** |
| 大整数被四舍五入（8） | 走原生 `JSON.parse`，float64 精度丢失 | **默认 BigInt 保真**，显示与复制都不失真 |

**护城河 = 正确性与可信赖**。这不是一句口号，它必须在 UI 上看得见：
- `✓ N big-ints exact` 徽章 —— 别处会把 `136986234663732436` 悄悄变成 `...430`
- `⚠ N duplicate keys — last value shown` —— JSON 规范规定后者覆盖、其余静默丢弃，只有我们标出来
- 状态栏常驻 `big integers kept exact · no ads · no telemetry`

**UI 重做的分寸**：视觉可以全换，但**这三个信任信号必须在新设计里有同等或更强的存在感**。它们是产品存在的理由，不是装饰。

**设计语言（现有，可挑战）**："Quiet Precision" —— Linear 式冷静底色 + IDE 式结构栏/状态栏 + 信任文案。

## 2. 三个界面（Surface）

产品有三个物理上独立的界面，**优先级从高到低**：

| ID | 界面 | 触发方式 | 尺寸 | 优先级 |
| --- | --- | --- | --- | --- |
| **S1** | 接管页 | 访问任意 JSON 网址，页面被自动替换 | 整个标签页（全屏） | **P0 主战场** |
| **S2** | Popup | 点击工具栏扩展图标 | **固定宽 328px**，高度自适应（上限约 600px） | **P0 入口** |
| **S3** | Viewer 页 | popup 点"Format JSON →"后新开标签页 | 整个标签页 | P1 |

S1 与 S3 共用同一套渲染组件（工具栏 + 面包屑 + 结构栏 + 树 + 状态栏），差别只在 S3 顶部多一个粘贴输入区。**原型设计时应把它当同一个组件的两种装载环境。**

├──────────┬────────────────────────────────────────────┤

| 3 | 折叠按钮 | `⤢ Collapse all` ⇄ `⤡ Expand all` | 全展/全收 | 无嵌套时**自动隐藏** |

| 5 | 搜索框 | placeholder `Search keys & values` | 见 §4.6 | 右侧有 `/` 快捷键角标 |
| 6 | 下载 | `⤓` | 下载 .json | 图标按钮 |
| 7 | 主题 | `◐` / `☀` / `☾` | auto→light→dark 循环 | tooltip: `Theme: auto` |

| 9 | 元信息 | `1.2 MB · 8 keys` | 只读 | 等宽 |

### 4.3 面包屑

- 常态：`root`
- 点击任意节点后：`root › customer › email`
- 等宽字体，单行

### 4.4 结构栏（左侧）

- 标题：`STRUCTURE`
- 内容：顶层 key 列表，每项 = key 名 + 子节点数徽章
- 点击 → 滚动到该区块，目标行短暂高亮（约 900ms）
- 滚动时自动高亮当前所在区块（scroll-spy）
- **出现条件**：存在嵌套结构 **且** 顶层项 ≥ 3；否则整栏隐藏（扁平 JSON 不需要）

### 4.5 树视图

| 元素 | 规格 |
| --- | --- |
| 行号 | 左侧独立槽（gutter），等宽，弱色 |
| 缩进 | 每层 2 空格，建议有缩进引导线 |
| 折叠箭头 | `▾` 展开 / 折叠时旋转；叶子节点占位不可点 |
| key | 带引号，key 色 |
| value | 按类型着色：string / number / boolean / null 各一色 |
| **大整数** | 特殊高亮（`jk-precise`），tooltip: `kept as an exact integer — never rounded to a float` |
| 容器计数 | `8 keys` / `3 items`，跟在 `{` `[` 后 |
| 折叠预览 | 折叠时显示 `… }`，计数隐藏 |
| **hover 动作** | 行右侧浮出两个按钮：`⧉`（复制值/子树）+ `path`（复制访问路径如 `customer.email`） |

### 4.6 搜索

- 输入即搜（无需回车）
- 命中行高亮，**非命中行变暗**（dim）
- 计数 `1/2`，`↑` `↓` 跳转（Enter = 下一个，Shift+Enter = 上一个）
- 自动展开命中项的父级
- `/` 聚焦搜索框
- 当前命中项有独立高亮色（`jk-current`）
- 在 Raw 视图搜索会自动切回 Pretty

### 4.7 状态栏

从左到右：

| 元素 | 文案 | 条件 |
| --- | --- | --- |
| 有效性 | `● valid JSON` | 常驻 |
| **重复 key 警告** | `⚠ 2 duplicate keys: "a", "b" — last value shown`（超 4 个截断为 `…`） | 仅当存在重复 key |
| 节点数 | `1,234 nodes` | 树已构建时 |
| 类型统计 | `12 obj` `3 arr` `88 str` `14 num` `2 bool` `1 null`（数量为 0 的类型不显示） | 树已构建时 |
| 大文件提示 | `1.2 MB — large file, tree built on demand` | 仅大文件模式 |
| **信任文案** | `big integers kept exact · no ads · no telemetry` | **常驻，靠右** |

warning 的 tooltip 原文：`JSON spec keeps only the last value of a duplicated key; other viewers drop the rest silently`

### 4.8 闪现提示（flash）文案原文

`Copied valid JSON ✓` / `Copied value ✓` / `Copied subtree ✓` / `Copied path ✓` / `Downloaded ✓` / `Copy blocked` / `Select & ⌘C to copy`

### 4.9 状态穷举（★ = 现状缺失，本次必须补）

| 状态 | 现状 | 要求 |
| --- | --- | --- |
| 正常 | 树视图 | — |
| **大文件（>1MB）** | 默认进 Raw，状态栏提示；切 Pretty 才建树 | 保留该降级，但**降级理由要让用户看懂**，并给"仍要构建树"的明确入口 |
| **建树中** ★ | 无加载态，大文件切 Pretty 会卡住无反馈 | 需要加载态/骨架屏/进度 |
| **解析失败** | 仅一行 `Not valid JSON: {原始报错}` ★ | 见 §5.4 修复模式 |
| 空对象 `{}` / 空数组 `[]` ★ | 渲染成空树，无说明 | 需要明确空态 |
| **搜索无结果** ★ | 计数显示 `0`，全部行变暗，像坏了 | 需要明确的空结果态 |
| 单值 JSON（如 `42`、`"hi"`） | 单行渲染，结构栏隐藏 | 保持 |
| 剪贴板被拒 | `Copy blocked` / `Select & ⌘C to copy` | 保留 |
| 深/浅主题 × 4 种配色 | 已支持 | 每个界面至少出深浅两版 |

## 5. 新功能（本次新增，均为纯本地计算）

> 四组功能都锚定同一条线：**别家做不好的"正确性与可信赖"**。设计时按 P0 → P2 排优先级取舍。

### 5.1 查询与表格视图（P0，UI 分量最重）

**痛点**：搜索只能高亮，不能筛出子集；数组型 JSON（API 列表响应）用树看远不如表格。

| 需求 | 说明 |
| --- | --- |
| [F-101] 查询栏 | 输入 JSONPath（如 `$.users[*].email`）或 jq 表达式，结果**替换主视图**显示匹配子集 |
| [F-102] 查询结果计数 | `3 matches`，以及"回到完整视图"的明确出口 |
| [F-103] 查询语法错误态 | 就地提示错在哪，不清空已有结果 |
| [F-104] 表格视图 | 对象数组 → 表格：列 = 并集 key，行 = 元素；缺失字段显式标记（不是空白） |
| [F-105] 表格 ↔ 树互跳 | 点表格单元格 → 跳到树里对应节点 |
| [F-106] 表格不可用态 | 非数组/异构数组时，说明为什么不能表格化 |

**设计问题**：视图现在是 `Pretty | Raw | Min` 三态，表格是第四态吗？还是正交的另一维（因为 Table 只对数组有意义）？**请在原型里给出答案。**

### 5.2 Schema 与类型导出（P1）

**痛点**：开发者拿到 API 响应，下一步就是写类型定义 —— 现在得手抄。

| 需求 | 说明 |
| --- | --- |
| [F-201] 反推 JSON Schema | 从当前 JSON 生成 Schema，可复制/下载 |
| [F-202] 导出 TypeScript 类型 | 生成 `interface`/`type`，可复制 |
| [F-203] Schema 校验 | 贴入一份 Schema，校验当前文档，**错误定位到具体节点**（树上标红） |
| [F-204] 类型推断的诚实标注 | 可空/联合类型/空数组等推断不确定处**必须标出来**，不许假装确定 —— 这是"可信赖"的一部分 |

**IA 提示**：这是"导出"族（和现有的 Copy / Download 同族），考虑收纳进统一的导出菜单。

### 5.3 历史记录与 Diff（P1）

**痛点**：调接口时"刚才那个响应变了什么"是真痛点。

| 需求 | 说明 |
| --- | --- |
| [F-301] 历史列表 | 最近查看/粘贴过的 JSON（存本地 storage），显示时间 + 大小 + 首行摘要 |
| [F-302] 历史入口 | popup 内显示最近 3 条；查看器内可开完整历史面板 |
| [F-303] 删除与清空 | 单条删除 + 一键清空，**必须有**（本地隐私工具的基本尊重） |
| [F-304] 容量上限 | 明确上限（如 20 条 / 5MB），超出淘汰最旧，UI 要说明 |
| [F-305] 两份 JSON Diff | 左右分栏对比，标出新增/删除/修改 |
| [F-306] Diff 入口 | 从历史选两条对比，或粘贴第二份 |
| [F-307] Diff 空结果态 | 两份完全相同时明确说"无差异"，不是空白 |

### 5.4 修复模式与虚拟滚动（P0 / P2）

| 需求 | 优先级 | 说明 |
| --- | --- | --- |
| [F-401] 可视化报错 | **P0** | 解析失败时显示原文 + **定位到出错行列**（高亮那一行），而不是只给一行文字 |
| [F-402] 修复建议 | P1 | 常见错误给一键修复：尾逗号、单引号、缺引号的 key、注释（JSONC 已容错但可提示）、被截断的 JSON |
| [F-403] 修复预览 | P1 | 修复前后对比，用户确认后才应用 —— 不许静默改用户数据 |
| [F-404] 虚拟滚动 | P2 | 极大文件（>10MB）树视图不卡。UI 影响：滚动条语义、行号连续性、搜索跳转定位 |

> [F-401] 优先级高的原因：解析失败是**用户最挫败的时刻**，现在的一行报错是整个产品体验最弱的地方。

## 9. 开放问题（原型阶段可先假设，落地前需确认）

1. **查询语法选 JSONPath 还是 jq？** 前者简单常见，后者强大但学习成本高。原型可先按 JSONPath 画，但需为"语法切换"预留位置。
