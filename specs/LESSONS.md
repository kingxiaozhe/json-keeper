# LESSONS —— 踩坑与方向性取舍记录

> 开发时必读。只记「下次会因此改变做法」的东西，不记流水账。

## L-001：防护网要锁"装配"，不能只锁"零件"（T-001）

**踩坑**：第一版基线写了 58 条测试，覆盖 `jsonbig.js`（解析）、`normalize`、`esc`/`escAttr`，自认扎实。对抗审查实测打脸：

- 把 `core.js` 的 `esc(key)` 改成裸 `key`（**真 XSS**，JSON key 直进 innerHTML）→ **58 条全绿**
- 模拟拆分时 `manifest.json` 漏登记 `util.js`（**接管 100% 失效且无报错**）→ **58 条全绿**

**根因**：网保护的面积（jsonbig 的解析层）和拆分要动的面积（core.js 的 DOM 层 + 加载拓扑）**几乎不重叠**。锁了函数本身，没锁函数**有没有被调用**。

**更隐蔽的教训 —— 我的变异测试骗了我**：我当时改的是 `escAttr` 的**定义**，而测试正好从源码里抠那两行来测 —— 测的和改的是同一个东西，红是必然的，证明不了任何事。

**下次怎么做**：
- 变异测试必须改**调用点**、改**配置**、改**拓扑**，不能改被测函数自己的定义。
- 写完网先问："接下来那个改动最可能怎么翻车？这张网会红吗？"—— 而不是"我覆盖了几个函数"。
- 加载顺序/配置登记这类拓扑问题，测试必须**读配置文件驱动**（`loadPerManifest`），硬编码加载顺序等于把头号风险变成盲区。

## L-009：我连续三次用"自己能抓的变异"证明网有效（T-001 / T-002 / T-003）

**这是本项目最贵的一条教训，因为它重复了三次。**

| 任务 | 我的自测 | 对抗审查的实测 |
| --- | --- | --- |
| T-001 | 改 `escAttr` **定义** → 红 → "网有效" | 改 `esc(key)` **调用点**（真 XSS）→ **58 全绿** |
| T-002 | 改 manifest / 调用点 → 红 → "网有效" | `expandTo` 空实现、折叠不隐藏子行… → **5/6 存活** |
| T-003 | 改 `setChip`/`renderStatus` → 红 → "网有效" | `applySort` 空操作、Collapse all 死键… → **10/12 存活** |
| T-009 | 6 个靶全打 `sayNoHits` 的调用路径 → "6/6 全抓" | DOM 顺序 / mount 初始态 / `findN` → **3/11 存活** |

**共同的病**：我挑的变异，恰好是我的测试覆盖的那部分。**这不是验证，是循环论证。**

**T-009 补充了一个可操作的判据**：三个存活的靶全在**我的测试从没读过的属性**上 —— 13 条测试对 `findN` 一条断言都没有，对 `children` 的**顺序**一条都没有。所以选靶前先列**这个模块对外可观察的属性有哪些**（文本、顺序、可见性、时序），再看每根轴上有没有断言。**没有断言的轴 = 必然存活的靶**，不必等审查来告诉我。

**下次怎么做**：
- 变异测试的靶子必须**先于**测试选定 —— 从"这个改动最可能怎么坏"出发列清单，再看网抓不抓得住。**不能反过来**。
- 靶子要打**调用点、配置、装配、状态同步**，不打被测函数自己的定义。
- 报数字时报**全部靶子的战果**，不报挑出来的那几个。3/10 是数字，"5 个变异全抓"是自我安慰。

## L-012：「变异我自己的断言」是循环论证，证明不了那个性质重要（T-005）

T-005 我写了一条测试断言「manifest 里 tokens.css 必须排在 viewer.css **之前**，否则 var() 全落空、界面无色」，然后变异顺序 → 测试红 → 宣布「7/7 全抓」。

**理由是错的，我自己查出来的**：CSS 自定义属性在**计算值阶段**解析，`var()` 照样看得到后加载的 `:root` 定义 —— 顺序只在同名 token 被覆盖时才有意义，而 `viewer.css` 定义了 **0 个** token。**颠倒顺序什么也不会发生。**

于是那个「抓到」是纯循环：**测试断言 X → 我变异 X → 它必然红**。这只证明测试在自洽，完全没证明 X 重要。

这是 L-009 的新变种，而且更隐蔽 —— L-009 是「靶子挑得偏」，这条是「靶子打中了，但那个靶子本身没意义」。

**下次怎么做**：
- 变异之后多问一步：**「这个变异在真实世界里会让用户看到什么？」** 答不上来，说明这条测试守的是幻觉。
- 顺着"这个变异真的会坏吗"去查规范/实测，**而不是看测试红不红**。测试红只说明测试和代码一致。
- 修法不是删掉测试，是**换成守真会出事的那个性质**：这里改成「tokens.css 必须被装载」（漏装才真的无色），并留一条「只有一个表定义 token」守住"顺序不重要"的前提 —— 哪天它破了，顺序会重新变成雷。

## L-014：与其留一个"为将来预留"的防御，不如把洞填掉（T-008）

popup 的状态行原本 `innerHTML = esc(e.message)`。测试断言"报错文案不含 `<img`" —— **恒真**：那个输入的报错是 `Expected ':' at position 31`，压根不含原文，删掉 esc 照样绿。

查下去发现更根本的事实：**jsonbig 的全部报错里，只有 `Invalid number: <字面量>` 引用原文，而数字字面量只含 `0-9 e E + - .`，装不下 `<img`**。所以 esc 在防一个**当时不存在的洞**。

我最初的处置是"保留 esc + 在测试里写实话（今天验不到，feature 5 的 F-401 会让它成为真攻击面）"。**那是个次优解** —— 它留下一个谁也说不清有没有用的防御，下一个人还得重新查一遍。

**更好的处置：改 `textContent`。** 没有解释 HTML 的地方，就没有注入面；esc 随之成为死代码，删掉。F-401 真要富文本高亮时再换回来，那时 esc 是有明确理由的。

**判据**：防御 vs 消除。**能消除攻击面就别防御它** —— 防御要靠人记住它为什么在，消除不用。

## L-010：手写 DOM 桩测不了 innerHTML 建的 UI —— 承认它，别灌水（T-003）

产品用 `innerHTML` 字符串建 UI，桩不解析 HTML → `$('[data-act="fold"]')` 拿到幻影元素 → **凡依赖工具栏按钮的行为一律测不到**。我往桩里补了假 chrome（复活 `applySort`）、补了 `textContent` 聚合（复活搜索匹配），10 个变异也只从 2 个提到 4 个被抓。

**决定：不引 jsdom。** 这个扩展的立身之本是零依赖、无远程代码、审核可读 —— 为测试拉进几十个传递依赖与之冲突，而 `testing.md` 本来就把 DOM 层划给了手动冒烟清单。

**关键动作不是灌水，是止损**：把测不到的 6 条**逐条写进冒烟清单第 11–15 项**（每条对应一个实测存活的变异，不是假想），并在 `mount.test.mjs` 头部写明"这层没有网"。

**一个假装有网的测试，比没有测试更危险** —— 后者你会小心，前者你会放心。

## L-011：测试标题写了没验的事 = 撒谎（T-003b）

两个实例，都是我写的：

1. 标题「只展开目标的祖先链，**不动兄弟**」—— 断言只查了目标可见。而底下正好藏着 `_collapse` 的泄漏 bug（展开外层会把内层仍折叠的块一起露出来）。
2. 标题「Raw 视图下跳转会先切回 Pretty」—— **从头到尾没有触发跳转**，只断言了两个前提。删掉 `setView("pretty")` 整行，它照样绿。

**更坏的是第 2 条的断言与生产相反**：它断言 `railEl.hidden === false`，而 `jk:view=raw` 新开时 `rail.render` 一次都不会调（`renderTree` 只在 `setView("pretty")` 里调）→ 生产里是 `true`。测试读到 `false` 纯粹因为桩不解析 SHELL 的 `hidden` 属性、`El.hidden` 默认 false。

**下次怎么做**：
- 写完断言，回头读一遍标题：**标题声称的每个性质，都有对应的断言吗？** 没有就改标题或补断言。
- 「目标可见了」不等于「只有目标可见」—— 一个把整棵树全展开的实现同样能让目标可见。**正向断言必须配反向断言。**
- 桩给出的默认值（`hidden` 默认 false、`querySelectorAll` 返回 `[]`）会伪装成"产品行为"。断言前先问：**这个值是产品设的，还是桩的默认值？**

## L-006：拆分会让"命名空间存在"这个守卫失效（T-002）

**踩坑**：`content.js` 靠 `if (!window.JK) return;` 决定动不动宿主页。拆分后 `util.js` 一加载就建了 `window.JK` —— **守卫从此恒真**，哪怕 `tree.js` 根本没加载。

**失败链（已实证）**：manifest 漏登记 `tree.js` → 守卫放行 → `mountViewer` 走到 `store.get`（**异步**）就 `return true` → content.js 据此 `body.textContent = ""` **清空整个页面** → 随后回调里 `JK.tree.build` 抛 TypeError，content.js 接不住 → **原页面已毁、树区空白、用户零提示**。

**最讽刺的**：我在 `util.js` 和 `core.js` 的注释里**两次精确诊断了这个洞**（"content.js 的 !window.JK 守卫会直接放行"），却没去改 `content.js`。**写下诊断 ≠ 修好问题**。

**下次怎么做**：
- 把命名空间拆开时，**所有"这个全局存在吗"的守卫都要重新审一遍** —— 它们的语义被拆分悄悄改变了。
- 更根本的：让**入口函数自己检查依赖**（`mountViewer` 开头 `if (!JK.tree) return false`），而不是让调用方猜。契约是"false = 别动页面"，那它就该在承诺之前把话说死。
- 注意**异步回调**会让"返回成功"和"真的成功"分家：`return true` 在建树之前，调用方据此做了不可逆的事。

## L-007：测试桩的保真度决定了网的有效性（T-002）

`_dom.mjs` 的 `querySelector()` 每次返回**新元素** → caret 上挂的 `_collapse` 立刻被丢弃 → `expandTo`/`collapseAll` 在每条测试里**必然空转**。后果：对抗审查跑了 6 个变异，**5 个存活**（expandTo 改空实现、折叠不隐藏子行、byPath 改 last-wins…全都 102 全绿）。

这是 **L-001 的小范围重演**：搬走的老代码网很牢，**新长出来的 API 一条网都没有**。

**下次怎么做**：桩改成按 (元素, 选择器) 记忆化后，同样的变异 4/5 被抓。**写桩时先问："被测代码把状态挂在哪？我的桩留得住吗？"** 留不住的话，所有依赖那个状态的断言都是空转 —— 而它们看起来全绿。

## L-008：测试抓不到的变异，可能是死代码（T-002）

`byPath` 的 `if (!byPath.has(apath))` 守卫，改成 last-row-wins 后测试仍全绿。查证后确认：闭合行不传 apath、每个节点 apath 唯一，**两行永不撞 key —— 守卫是空转的**。

删掉它，而不是为了"让变异被抓"去补一条测试。**一个自己的测试都分辨不出差异的"守卫"，就是在骗读代码的人。**

## L-015：桩的失真会**两头**骗人 —— 也会凭空造出不存在的 bug（T-009）

前八条教训里，桩的失真一直只以一种形式出现：**掩盖真 bug**（L-007：留不住状态 → 断言空转）。T-009 撞见了反向的那一半。

反复 mount 时 `.jk-nohits` 累积到 4 条，看着就是一个货真价实的泄漏。**它不存在**：真实浏览器执行 `rootEl.innerHTML = SHELL` 会销毁整棵旧子树，`querySelector` 拿到的是全新元素；而我的桩清了 `children` 却**没清记忆化缓存 `_q`**，于是把"每次都是新元素"演成了"同一个元素被插了 5 次"。

**没查就"修"的代价**：我会给 `search.js` 加一段清理逻辑，为一个不存在的 bug 增加复杂度 —— 而且它会一直绿，因为它"修好"的是桩。

**下次怎么做**：桩里看到异常现象，先问**「真实浏览器在这一步会做什么？」**再问「产品哪里写错了」。修桩和修产品是两件事，选错方向的成本是往产品里塞死代码。

## L-016：同步的假 storage 会盖住真实的异步窗口（T-009）

`noHits.hidden = true` 这行初始化，在假 storage 下是**冗余**的 —— `core` 的 `store.get("jk:view", cb)` 回调会 `setView` → `onViewChange` → 把它设回 true，所以删掉它测试全绿（对抗审查实测存活）。

**但真实 `chrome.storage` 是异步的**：回调到达之前，`hidden` 的默认值 false + `.jk-nohits` 的 padding/border/background = **每个 JSON 页一打开，树顶挂一条空白灰条**。这与 T-007 的主题闪白是同一个形状 —— 都藏在"回调还没到"的那一帧里。

`installChrome` 早就有 `{async:true}` 开关（T-003 加的，理由正是"真实 Chrome 是异步的"），我没用。

**下次怎么做**：凡是**在 storage 回调里被赋值**的状态，都要用 `{async:true}` 补一条「回调到达之前是什么样」的断言。同步送达是测试的便利，不是产品的现实 —— **默认值会在那一帧里被用户看见**。

## L-002：`node --test tests/` 是错的（T-001）

Node 24 会把目录当模块解析并报 `MODULE_NOT_FOUND`。正确写法 `node --test "tests/*.test.mjs"`，**引号不能少**（否则 shell 先展开）。`.claude/rules/testing.md` 里最初写错了，已修。

## L-003：文本位置的引号是无害的（T-001）

写 XSS 测试时误判过一次：断言 `!html.includes('onmouseover="alert(1)"')` 失败，以为代码有洞。实际上那串出现在 `<span class="jk-key">"a" onmouseover="alert(1)"</span>` —— **文本位置**，浏览器当文本渲染，无害。

这正是 `esc`（不转引号）与 `escAttr`（额外转引号）分成两个函数的原因。**写安全测试时要盯准攻击面在哪，别把无害命中当漏洞** —— 否则会为了让测试变绿去"修"没坏的代码。

## L-004：不改造被测代码来迁就测试（T-001）

`esc`/`buildTree` 等是 `core.js` 的内部件，没经 `window.JK` 导出。为了测它们，**没有**去改源码加导出（那等于在没有网的情况下先动刀）。做法是加载时把导出行替换成更宽的导出，跑的仍是同一份源码。

第一版用正则抠单行 `const` 更脆：`esc` 改成多行、提取共享常量、或改成 `function` 声明，都会让**行为保持的重构误报红**。已换成替换导出行的方式。

## L-005：现状缺陷要锁住并标注，不要回避（T-001）

`[1e]` → `NaN` → Copy 出 `[null]`；`[1e999]` → `Infinity` 同理 —— **v0.8.0 线上就在静默改用户数据**，与产品"不偷改你的数据"的核心承诺直接冲突。

基线里用 `[现状缺陷]` 标注照实锁住。这样 feature 5 修它时**必须显式改掉这些断言** —— 那个动作就是让改动被看见的机制。防护网锁的是"今天它就是这么干的"，不是"它应该这么干"。

**待裁决**：这个缺陷不在 53 个任务的任何一条里（见 `specs/5.repair-mode-and-virtual-scroll/requirements.md` 的待裁决 1）。当前决定：并入 feature 5。
