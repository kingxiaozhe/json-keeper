# 修复模式与虚拟滚动 — 需求规格

## 概述

把"解析失败"从一行冷冰冰的报错，变成能定位、能修、能预览的体验；并让极大文件的树视图不卡。

## 项目信息

- 项目名: json-keeper
- 架构类型: 单体前端应用（Chrome MV3 扩展 · 零依赖原生 JS · 无构建）

## 需求版本

| 日期 | 版本 | 说明 |
| ---- | ---- | ---- |
| 2026-07-16 | v1 | 初始需求（源: `specs/docs/PRD-json-keeper-ui.md` §5.4） |

## 用户故事

- 作为**贴进一段坏 JSON 的人**，我想要看到到底是第几行第几列出的问题，以便我不用自己数括号。
- 作为**从日志里复制出半截 JSON 的人**，我想要一键把常见毛病（尾逗号、单引号）修掉，以便我不用手工改。
- 作为**在意数据的人**，我想要修复前先看到会改成什么样，以便我确认它没改坏我的数据。
- 作为**打开 50MB JSON 的人**，我想要树视图能滚而不是卡死，以便我能真的用它。

## 功能需求

1. [F-401] **可视化报错**：解析失败时显示原文并定位到出错**行列**，高亮那一行。
2. [F-402] 修复建议：常见错误给一键修复 —— 尾逗号、单引号、无引号 key、被截断的 JSON。
3. [F-403] 修复预览：修复前后对比，**用户确认后才应用**。
4. [F-404] 虚拟滚动：极大文件（> 10MB）树视图可流畅滚动。
5. [F-405] 虚拟滚动下**行号保持连续正确**，且"复制路径"与搜索跳转定位仍准确。

## 非功能需求

- **性能**（对抗审查下调 —— 原写"50MB 首屏 < 2s"**不可能达成**）:
  - **滚动**: 10MB JSON 在树视图下滚动帧率 ≥ 30fps。这是虚拟滚动**能**解决的部分。
  - **首屏**: **不承诺具体秒数**。虚拟滚动只砍 DOM 节点数，砍不掉两段同步阻塞：① `jsonbig.js` 是逐字符 JS parser（`string()` 里 `s += c`），50MB 光解析就是秒级；② `buildModel` 要为每行生成 `html` 字符串。**这两段本轮 5 个 feature 里没有任何任务碰**。首屏承诺需要独立的解析/建模优化 feature，见开放问题。
  - **搜索**: 击键响应 ≤ 150ms（防抖后），复杂度必须 O(n) 而非 O(n×m)。
- **安全**: 错误原文片段是不可信数据，渲染一律 `esc()`；**修复不得静默改用户数据**（F-403 是安全要求，不只是体验）。
- **兼容性**: 零依赖 —— 虚拟滚动自研，不得引入 npm 包或 CDN。
- **体积**: 新增 `repair.js` / `vlist.js` 各 ≤400 行。

## 验收标准

- [ ] [AC-401] 喂 `{"a":1,\n"b":}` → 错误提示含**行号与列号**（如 `第 2 行第 6 列`），且该行在原文视图中被高亮。
- [ ] [AC-402] 喂含尾逗号的 `{"a":1,}` → 给出"尾逗号"修复建议（注：`jsonbig.js` 已容错尾逗号，故此例应能直接解析 —— 本项验收改用**不可容错**的组合，见下）。
- [ ] [AC-403] 喂 `{'a':1}`（单引号）→ 给出"单引号 → 双引号"修复建议。
- [ ] [AC-404] 喂 `{a:1}`（无引号 key）→ 给出"key 缺引号"修复建议。
- [ ] [AC-405] 喂 `{"a":[1,2` （被截断）→ 给出"补齐缺失的 `]` `}`"修复建议。
- [ ] [AC-406] 点击任一修复建议 → **先出现前后对比预览**，未确认前原文不变；确认后才应用。
- [ ] [AC-407] 修复预览中的差异**可见**（改了哪几处一目了然）。
- [ ] [AC-408] 打开 10MB JSON 并切 Pretty → 树可滚动，滚动不卡死（人工感知 + DevTools 帧率抽查）。
- [ ] [AC-409] 虚拟滚动下滚到第 10000 行 → 行号显示正确且连续（非从 1 重新计数）。
- [ ] [AC-410] 虚拟滚动下用搜索跳转到某命中项 → 正确滚动到该行并高亮。
- [ ] [AC-411] 虚拟滚动下 hover 任一行点 `path` → 复制出的访问路径正确。
- [ ] [AC-412] 修复后的文本经 `JSONBig.parse` 可解析，且大整数仍保真。
- [ ] [AC-413] 错误原文片段含 `<script>` 时以纯文本显示，无弹窗。
- [ ] [AC-414] 复跑防护网基线（`node --test tests/`）全绿。

## 依赖

- 前置 feature: `1.ui-shell-redesign`（模块拆分后的 `tree.js` 挂载点）、`4.history-and-diff`（**复用其 `jsondiff.js` 做修复预览的前后对比**）。

## 开放问题

- **无设计稿**（Stitch 未生成本组功能的界面）。经人工确认：UI 由前端按 design.md 自行实现，不生成 UI 还原任务。
- 已确认（技术决策，非产品决策）：虚拟滚动下**行号必须保持连续** —— 否则"复制路径"与搜索跳转的心智模型断裂（F-405）。
- 遗留（不阻塞）：`jsonbig.js` 当前已容错 JSONC 注释与尾逗号，故这两类"错误"不会走到修复模式。修复模式覆盖的是**真正解析不了**的输入。

### ⚠ 待人工裁决 1：v0.8.0 线上存在静默改数据的缺陷（超出本 feature 范围）

对抗审查发现、**已实测确认**（`node` 直接跑 `jsonbig.js`）:

| 输入 | 实际行为 | Copy JSON 输出 |
| --- | --- | --- |
| `[1e]` | `isFloat` 分支 → `Number("1e")` = **NaN**，**不报错**，解析"成功" | `[null]` |
| `[1e999]` | → **Infinity**，不报错 | `[null]` |

后果链: `content.js` 判定合法 → **接管页面** → 树里显示 `NaN` → 用户点 Copy → 拿到 `[null]`。**这是静默改用户数据 —— 与本 feature「修复不得静默改用户数据」是同一条主张的反面，也与产品「big integers kept exact」的护城河叙事直接冲突。**

它**永远走不到修复模式**（因为压根不抛错），AC-412 也验不到。**不在本轮 5 个 feature 的任何范围内。**

请裁决: ① 并入本 feature（+1 个任务，改 `jsonbig.js` 的 `number()` 对 NaN/Infinity 报错或标警告）② 单开一个修复 feature ③ 本轮不处理，记录在案。

### ⚠ 待人工裁决 2：50MB 首屏承诺被移除

原 NFR "50MB 首屏 < 2s" 经审查判定**不可能达成** —— 虚拟滚动只砍 DOM 节点数，砍不掉 `jsonbig.js` 的逐字符解析（`s += c`）与 `buildModel` 的逐行 html 字符串生成，两段都是同步阻塞，且本轮无任何任务碰它们。已下调为"不承诺具体秒数"。

请裁决: 是否需要独立的「解析/建模性能」feature？若需要，它应排在 feature 5 之前还是之后？
